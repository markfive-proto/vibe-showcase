Best practices for structuring a monorepo?

Help Solved · Started 2 hours ago · 23 replies
AK
Alex Kim 2 hours ago

Hey everyone! I'm working on a project that has grown to include a frontend (Next.js), a backend (Go), a shared types package, and infrastructure as code. We're 4 developers and things are getting messy with separate repos.

Has anyone migrated to a monorepo successfully? I'm evaluating Turborepo, Nx, and Bazel. Would love to hear what's worked for you, especially around:

  • CI/CD pipeline setup
  • Managing shared dependencies
  • Build times as the repo grows
JM
Jamie Martinez Accepted Answer 1 hour ago

We made this exact switch about 6 months ago. Went with Turborepo and it's been great. Here's what I'd recommend:

1. Start with Turborepo — it's the easiest on-ramp and handles caching really well. Build times dropped 60% for us with remote caching.

2. Shared packages — create a /packages directory for shared types, configs, and utils. Use TypeScript project references.

Happy to share our setup if you want more details!
